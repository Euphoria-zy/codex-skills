# Coding Standards

## 核心原则

在满足明确需求、正确性和必要安全性的前提下，选择总复杂度最低的实现。

极简不是追求最少行数，而是减少概念、状态、分支、文件、依赖和调用层级。代码必须直接解决当前问题，不为假想需求预建设施。

新增的每个文件、类、函数、字段、分支和配置都必须对应以下至少一项：

1. 明确需求或验收标准；
2. 已存在的真实调用契约；
3. 可复现问题或失败测试；
4. 已确认的安全、性能或运行约束。

无法指出依据时，删除。不得用“最佳实践”“更完整”“更灵活”“以后可能需要”代替证据。

## 工作顺序

1. 阅读需求、项目规范、相关代码、直接调用方和现有测试。
2. 用一句话确认实现范围，并明确不做什么。
3. 把需求转成可验证的验收项；不要给简单任务附加设计流程。
4. 优先修改现有代码，选择最短、最直接、最符合现有项目结构的路径。
5. 只实现验收项需要的行为，并进行与风险相称的验证。
6. 交付前删除不能证明必要的代码。

需求不明确且不同理解会显著改变实现时，先确认；能够从现有代码和契约确定时，直接执行。不要用兼容层掩盖不确定性。

## 复杂度准入

增加抽象、helper、wrapper、parser、配置、依赖、缓存、重试或兼容逻辑前，必须同时满足：

- 当前存在真实需求或真实问题；
- 直接实现或修正源头不能更简单地解决；
- 新结构的职责和边界可以一句话说清；
- 新增复杂度小于它消除的复杂度；
- 可以通过测试或运行结果验证。

默认不新增：

- 只转发参数或改名的 wrapper；
- 只调用一次且没有独立业务含义的 helper；
- 没有当前使用方的基类、协议、工厂、注册表和扩展点；
- 为假想格式准备的 parser、alias 和 normalizer；
- 没有数据证明必要的缓存、并发控制和性能优化；
- 没有明确幂等性、终止条件和故障证据的重试；
- 用户未要求的兼容路径、降级逻辑和 feature flag；
- 仅为“分层完整”增加的 service、manager、adapter 或 repository。

少量清晰重复通常优于错误抽象。只有当前存在多个调用方、业务规则相同且应当共同变化时才抽取公共实现。

## AI 自我警戒

出现以下想法时停止编码并重新审视：

| 自我辩解 | 正确动作 |
|---|---|
| “以后可能复用” | 等真实的第二个使用方出现 |
| “这样更健壮” | 先指出已发生的失败和恢复策略 |
| “上游格式可能变化” | 先确认正式契约或真实样本 |
| “加一层更安全” | 明确该层独有且不可替代的安全不变量 |
| “这是最佳实践” | 说明它解决的当前问题，否则不做 |
| “顺便重构一下” | 移出本次范围 |
| “为了方案完整” | 以验收项而不是想象中的完整度为终点 |
| “多写一点以后省事” | 当前维护成本是真实的，未来收益是猜测的 |

禁止用代码展示知识。不要主动生成通用框架、预留接口、空扩展点、无调用方工具、假想错误分支或未启用配置。

## 函数、模块与抽象

- 函数以职责内聚、控制流清晰为准，不以短小为唯一目标。
- 能在一个清晰函数中完成的逻辑，不拆成多个跳转函数。
- 单次调用 helper 仅在隔离独立业务规则、显著降低复杂度或需要独立测试时保留。
- 不把简单条件包装成策略、规则引擎或插件系统。
- 不为少量类型差异建立继承体系；优先使用直接数据结构和普通组合。
- 不创建职责模糊的 `utils`、`helpers`、`common` 或 `manager` 作为杂物箱。
- 新文件必须形成清晰边界；仅放一个微小函数时优先留在现有模块。
- 抽象泄漏或需要大量适配时，优先修正接口，而不是继续增加适配层。
- 删除未使用代码、注释掉的代码和已经失效的入口。

## 数据、类型与契约

- 公共接口、跨模块数据和持久化数据必须使用确定类型。
- 在 Python 项目中，边界实体和共享实体优先使用 Pydantic `BaseModel`；字段使用 `Field` 明确必填、默认值和描述。
- Pydantic 模型用于稳定契约，不为单个函数的临时变量创建模型。
- `Any` 或无结构对象只允许短暂停留在未经校验的外部边界；完成校验后不得继续扩散。
- 已知字段直接通过属性访问，不用 `dict.get()` 或 `getattr()` 模糊真实契约。
- 枚举只用于真实、封闭并需要校验的值集合，不为一次性常量创建枚举。
- 外部数据只解析当前正式契约。真实存在多个版本时，在单一边界兼容并为每种格式提供样本测试。
- 不递归猜测任意 payload，不用大量字段别名容忍上游错误。

## 配置与依赖

- 只有需要由部署环境或用户动态修改的值才进入配置；固定规则使用常量。
- 配置必须有确定类型、默认值和校验，业务代码直接使用已校验结果。
- 在 Python 项目中使用 `BaseSettings` 管理环境配置时，不再通过 `or` 或重复 `None` 判断补默认值。
- 不增加当前逻辑未读取的配置项和开关。
- 优先使用项目已有依赖；新增依赖必须明显减少实现和维护成本。
- 不为一次简单调用封装第三方库；只有需要隔离不稳定边界或存在多个真实调用方时才建立适配层。

## 错误处理与外部副作用

- 在输入和外部系统边界校验数据，内部逻辑依赖已成立的契约。
- 只捕获能够恢复、转换或补充有效上下文的具体异常。
- `except Exception` 仅允许用于最外层执行边界，并保留原始异常原因。
- 不记录后又在同层重复抛出同一错误，不把失败伪装成空结果或成功结果。
- 不根据错误文本猜测复杂状态；重试必须基于明确错误类型和幂等保证。
- 写操作只执行明确授权的目标和动作；按实际风险加入确认、审计、回滚或幂等保护。
- 安全校验放在不可绕过且最接近副作用的单一边界，不在多层重复实现。
- 查询和命令必须参数化或受控构造，禁止拼接不可信输入。

## 修改与重构

- 只修改完成当前需求所需的最小范围，不做顺手重构。
- 保留无关现有行为和用户已有修改。
- 旧实现阻碍需求时，可以在目标边界内干净重写；不得扩大到无关模块。
- 替换旧实现后删除旧入口、旧分支和旧配置，不保留双轨逻辑，除非明确要求兼容。
- 优先删除、合并和直接调用，不把原有复杂度移动到更多文件。
- 重构必须降低认知成本；如果新增代码多于删除代码，应重新确认是否真的更简单。

## 测试与验证

- 缺陷修复先用测试复现问题，再写最小修复。
- 新行为围绕验收标准编写测试；测试行为，不测试私有实现细节。
- 不为不存在的边界情况编写代码和测试，也不为覆盖率制造无业务价值的测试。
- 测试数量和范围与风险相称：先运行最小相关测试，再按影响范围扩大。
- 配置、文档或纯机械修改使用相应的轻量验证，不强行制造单元测试。
- 没有实际运行验证，不得声称功能已通过、问题已修复或测试全部成功。

## Python 代码偏好

- 使用 Black 风格，`line-length = 140`。
- 公共函数、类属性、入参和返回值使用精确类型标注。
- 函数、类和模块 docstring 使用中文，说明职责或必要约束，不复述代码。
- 统一使用 f-string，禁止 `%s`、`%d` 和 `str.format()`。
- 一行不超过 140 字符且更清晰时可以写成一行；不要压缩复杂条件和副作用。
- 命名简洁、具体，避免 `universal_handler`、`normalized_data_processor`、`manager_factory` 等虚泛名称。
- 不在 `__init__.py` 批量暴露内容，除非它是明确的公共 API。
- 使用 UTF-8 编码并在 UTF-8 环境中验证。

## 交付前检查

- 每项修改是否直接对应需求、契约、真实问题或必要安全约束？
- 是否加入了假想需求、未来扩展或无证据的防御逻辑？
- 是否存在只转发的 wrapper 或无独立意义的单次 helper？
- 是否可以修正源头，而不是增加 parser、normalizer 或兼容层？
- 是否增加了没有数据支撑的缓存、重试、并发控制或依赖？
- 是否在多个层重复同一校验或错误处理？
- 是否修改了无关代码或覆盖了用户已有工作？
- 是否可以删除一层而仍满足全部验收项？
- 是否运行了足以支撑结论的验证？

任一项不满足，先精简再交付。

## Agent 交付要求

编码前用一句话说明实现范围和明确不做的内容。完成后只报告：

1. 实际修改；
2. 验证方式和结果；
3. 真实存在的风险或未完成项。

达到验收标准后立即停止。不要主动追加功能、重构、文档、配置或“最佳实践改进”。

**违反规则文字本身，就是违反本规范；不得以“遵循精神”为由绕过具体约束。**
